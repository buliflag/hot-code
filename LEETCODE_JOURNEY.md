### [215] 数组中的第K个最大元素 (Kth Largest Element in an Array)
- **核心思想**：**选秀淘汰赛 (Quick Select)**。利用快排分区思想，每次只保留目标所在的那个半区。
- **解法套路**：快速选择、堆排序（海量数据 VIP 房）。
- **💡 我的特定卡点**：
  - **随机选择与交换**：明白了随机化是为了防“有序数组”导致的效率退化；而“交换到最右侧”是为了**临时泊车**，避免在分区扫描时干扰基准值。
  - **原地修改 (In-place)**：理解了算法通过直接修改原数组（交换元素）来达到 $O(1)$ 的空间效率。
  - **步法逻辑**：掌握了“边防官 i”和“侦察兵 j”的模型——j 负责找人，i 负责守住边界并接收 j 换回来的小弟。
- **🔑 关键转折点**：意识到代码中的“随机选”其实是“随机换”；理解了分区本质是物理上的“大挪移”。
- **Go 实现细节**：
  - `nums[i], nums[j] = nums[j], nums[i]` 的原地交换是 Go 处理此类算法的标配。
  - `partition` 函数返回的 `i` 正是基准点最终的“坑位”。
- **面试表现点**：能清晰解释为什么 $O(n)$ 是通过不断“丢弃”半个搜索区间实现的。
- **关联题目**：[973] 最接近原点的 K 个点, [347] 前 K 个高频元素

### [208] 实现 Trie (前缀树)
- **核心思想**：**查字典索引**。共享前缀的单词走同一条路，节省空间且查找速度只与单词长度有关。
- **解法套路**：定义 `children`（路径）和 `isEnd`（标记单词终点）。
- **💡 我的特定卡点**：
  - **字符集扩展**：理解了 `[26]*Trie` 虽然快但在处理中文/多语种时不够灵活，此时应使用 `map[rune]*Trie`。
  - **业务应用**：思考了前缀树在“敏感词过滤”中的应用，理解了这是一种在长文本中进行“前缀匹配扫描”的过程。
- **🔑 关键转折点**：将 `Insert` 类比为“铺路”，将 `isEnd` 类比为“在路终点插红旗”，将 `Search` 类比为“顺着路走看有没有旗”。
- **Go 实现细节**：
  - Go 结构体嵌套实现：每个节点其实也是一棵完整的 Trie 树。
  - `StartsWith` 逻辑：只查路通不通，不关心有没有红旗。
- **面试表现点**：能讨论数组实现与 Map 实现的权衡（空间 vs 灵活性），并能联想到 AC 自动机作为进阶优化方案。
- **关联题目**：[211] 添加与搜索单词, [648] 单词替换

### [207] 课程表 (Course Schedule)
- **核心思想**：**通关任务与死锁**。利用“拓扑排序”检测图中是否存在环。
- **解法套路**：Kahn 算法（入度表 + 邻接表 + 队列）。
- **💡 我的特定卡点**：
  - **邻接表反向存储**：理解了 `adjacency[pre] = append(..., cur)` 是为了在修完 `pre` 时能精准定位受影响的 `cur`，而不是暴力遍历。
  - **入度计数 (Indegrees)**：将其类比为“门上的挂锁”，只有锁的数量减到 0 时，门才能打开（课程才能入队）。
- **🔑 关键转折点**：理解了“入度减 1”就是“拆掉一把锁”的过程；明确了拓扑排序的起点必须是入度为 0 的节点。
- **Go 实现细节**：
  - 使用 `make([][]int, n)` 构建邻接表。
  - 使用 `count` 计数或结果数组长度来判定是否修完了所有课。
- **🛠️ 拓扑排序万能模板 (Kahn 算法)**：
  1. **统计入度与邻接表**：`indegrees`, `adjacency`。
  2. **队列触发**：将所有 `indegrees == 0` 的节点入队。
  3. **循环拆锁**：出队一个节点，将其指向的所有后续节点入度减 1；若减至 0 则入队。
- **🔍 另一种思路：DFS 染色法**：
  - **类比**：排雷探路。
  - **三色标记**：0-未访，1-正在访（发现环的关键），2-已访完（安全）。
  - **面试权衡**：BFS 更好写、更好解释；DFS 更简洁，但需注意递归栈。
- **面试表现点**：能熟练写出拓扑排序三部曲，并能解释 BFS 和 DFS 两种检测环的方法及其优劣。
- **关联题目**：[210] 课程表 II, [802] 找到最终的安全状态

### [200] 岛屿数量 (Number of Islands) & [695] 岛屿最大面积
- **核心思想**：**沉没大法**。每发现一块陆地，就用 DFS 把与之相连的所有陆地“推平”变成水（1 变 0），防止重复计数。
- **解法套路**：
    - **200 题**：只求数量，DFS 负责“推平”即可。
    - **695 题**：求面积，DFS 负责“汇报”面积（`1 + dfs(四周)`）。
- **💡 我的特定卡点**：
    - **方向完整性**：在写 DFS 时，务必检查“上下左右”四个方向是否都走到了，避免漏掉（如 `row+1` 漏写成 `row-1`）。
    - **原地修改**：明白 `grid[i][j] = '0'` 的操作替代了 `visited` 数组，节省了空间。
- **🔑 关键转折点**：将 DFS 类比为“消灭证据的工程队”；将面积计算类比为“向上级汇报人口的村长”。
- **Go 实现细节**：
    - 处理 `[][]byte` (200题) 与 `[][]int` (695题) 时注意类型细节。
    - 在 Go 中定义匿名 DFS 函数时，需要先声明再赋值，才能进行递归。
- **关联题目**：[1254] 封闭岛屿数量, [827] 最大人工岛, [463] 岛屿周长

### [46] 全排列 (Permutations)
- **核心思想**：**填空游戏与橡皮擦**。利用回溯算法（Backtracking）遍历所有可能的选择。
- **解法套路**：`for` 循环横向选择 + `backtrack()` 纵向深入 + `path[:len-1]` 撤销回溯。
- **💡 我的特定卡点**：
  - **递归的暂停与继续**：理解了 `backtrack()` 像进入“多层梦境”，当一层梦醒（递归返回）后，会回到上一层暂停的位置，继续执行“擦除”动作。
  - **决策树模型**：通过“画树”明确了 `for` 循环是平级的并列尝试，而 `backtrack` 是向下的深度探索。
- **🔑 关键转折点**：将回溯类比为“带橡皮擦的 DFS”；理解了撤销操作是为了“恢复现场”，好让下一次并列选择在干净的环境下开始。
- **Go 实现细节**：
  - **切片深拷贝**：`res = append(res, temp)` 之前必须用 `copy`，否则结果集里的所有 path 都会指向同一个底层数组，最后变成一堆重复值。
  - **状态追踪**：使用 `used` 数组记录哪些数字已被占用。
- **面试表现点**：能清晰画出决策树，并对比回溯与普通 DFS 的区别（状态恢复）。
- **关联题目**：[47] 全排列 II (含重复数字), [78] 子集, [77] 组合

### [78] 子集 (Subsets)
- **核心思想**：**单向旋转门**。每一位元素都面临“选与不选”的抉择，且为了避免重复（如 [1,2] 和 [2,1]），必须规定只能向后挑选。
- **解法套路**：`for` 循环起点由 `start` 变量控制，不使用 `used` 数组。
- **💡 我的特定卡点**：
    - **为什么不回退**：理解了 `i = start` 的逻辑，它像一个只能向前转的门，拿了 `1` 就不能回头看 `1` 之前的数，从而天然地去除了重复组合。
    - **收集结果的时机**：全排列是“触底”才收集（叶子节点），子集是“每一步”都收集（所有节点）。
- **🔑 关键转折点**：将子集类比为“超市货架取货”，你走过了一个货架（索引），就不能再回头拿之前的商品了。
- **Go 实现细节**：
    - 递归入口从 `backtrack(0)` 开始。
    - 递归内部 `backtrack(i + 1)` 确保下一次搜索从当前元素的下一个开始。
- **面试表现点**：能对比排列（Permutation）与组合（Combination/Subset）在搜索树结构上的本质区别。
- **关联题目**：[77] 组合, [39] 组合总和 (允许重复选择)

### [39] 组合总和 (Combination Sum)
- **核心思想**：**带预算的购物**。在组合的基础上允许重复选择同一个数，并通过“剪枝”提升搜索效率。
- **解法套路**：排序 + 回溯 + 剪枝。
- **💡 我的特定卡点**：
    - **剪枝的先决条件**：深刻理解了**必须先排序**。如果候选数组无序，提前 `break` 会导致漏掉后面的解。
    - **重复选择的实现**：通过递归时传递当前索引 `i`（而非 `i+1`）来实现“原地踏步”，从而允许无限次选择同一个数。
- **🔑 关键转折点**：将剪枝类比为“超市购物超支警告”，一旦发现当前商品加进去就超标，直接放弃当前及之后更贵的商品。
- **Go 实现细节**：
    - `sort.Ints(candidates)` 是剪枝的灵魂。
    - `if sum + candidates[i] > target { break }` 这一行让性能提升了几个数量级。
- **面试表现点**：能向面试官解释为什么“剪枝”比“普通返回（return）”更高效；能展示如何在复杂搜索树中通过排序来缩减分支。
- **关联题目**：[40] 组合总和 II (含重复数字), [216] 组合总和 III
- **💡 进阶技巧：[827] 建模与预处理**：
    - **核心直觉**：填海造陆连大桥。
    - **两步走流程**：
        1. **人口普查**：DFS 为岛屿编号（从 2 开始）并统计面积存入 Map。
        2. **连通模拟**：遍历水域 (0)，检查四周岛屿编号，利用 `map[int]bool` 去重后累加面积。
    - **特定卡点**：**必须去重**。若水域四周连着同一个岛，不加去重会导致面积重复计算。
    - **性能精髓**：将 $O(N^4)$ 的暴力搜索优化为 $O(N^2)$ 的“预处理+查表”。
- **💡 进阶技巧：反向图拓扑排序**：
  - 在 [802] 中，通过**反转所有箭头**，将寻找“出度为 0 且不入环的点”转化为标准的拓扑排序问题。
  - 核心直觉：从“终点”逆向拆锁，能被拆掉锁的点就是绝对安全的。



### [198] 打家劫舍 (House Robber)
- **核心思想**：**二选一决策**。每一间房都面临“偷（前前家+当前家）”还是“不偷（前一家）”的选择。
- **解法套路**：动态规划。从 $O(n)$ 空间的数组方案演进到 $O(1)$ 空间的变量滚动方案。
- **💡 我的特定卡点**：
    - **初始化困惑**：最初不理解为什么 `prev2, prev1` 初始化为 0 就能处理所有情况，习惯于手动处理 `len < 2` 的边界。
- **🔑 关键转折点**：
    - **虚拟边界**：意识到将 `prev2, prev1` 初始化为 0 相当于在数组前加了两个“虚拟空房子”，这让循环逻辑变得极其统一且简洁。
    - **多重赋值**：理解了 Go 的 `a, b = b, a+x` 语法糖在空间压缩中的妙用。
- **Go 实现细节**：
    - `prev2, prev1 = prev1, max(prev1, prev2+v)` 这一行代码实现了状态的同步更新。
- **关联题目**：[213] 打家劫舍 II (环形), [337] 打家劫舍 III (树形)

### [238] 除了自身以外数组的乘积 (Product of Array Except Self)
- **核心思想**：**左右夹击**。结果等于当前位置左侧所有数的积乘以右侧所有数的积。
- **解法套路**：双向遍历。第一遍存左积，第二遍用变量维护并乘以右积。
- **💡 我的特定卡点**：
    - **空间复杂度理解**：纠结于输出数组是否算 O(n) 空间，后明确算法面试中输出数组通常不计入额外空间。
    - **空积初始化**：理解了 res[0] 和初始 right 设为 1 是基于数学上的空积定义。
- **🔑 关键转折点**：
    - **原位利用**：意识到直接在输出数组上操作可以节省掉两个 O(n) 的辅助数组。

### [155] 最小栈 (Min Stack)
- **核心思想**：**辅助栈模式**。用一个同步运行的栈来维护“到目前为止的最小值”。
- **解法套路**：辅助栈。优化版仅在 val <= minTop 时入栈。
- **💡 我的特定卡点**：
    - **重复最小值处理**：理解了为什么 val == minTop 时也必须入栈，否则出栈会导致数据不一致。
- **🔑 关键转折点**：
    - **保镖类比**：将辅助栈类比为记录历史最小值的“随身保镖”，实现了 O(1) 的检索。

### [213] 打家劫舍 II (House Robber II)
- **核心思想**：**破环成链**。将环形结构的冲突拆解为两个线性子问题。
- **解法套路**：分别计算 `[1, n-1]` 和 `[0, n-2]` 范围的 198 题解，取最大值。
- **🔑 关键转折点**：意识到通过两次线性调用就能覆盖所有合法的“首尾不兼顾”场景，避免在复杂的环形状态中纠结。

### [169] 多数元素 (Majority Element)
- **核心思想**：**对拼消耗 (Boyer-Moore Voting Algorithm)**。利用“多数元素数量超过一半”的硬性条件进行抵消。
- **解法套路**：摩尔投票法。通过 `candidate` 和 `count` 两个变量记录存活者。
- **💡 我的特定卡点**：
    - **算法前提**：理解了该算法之所以成立，前提是题目保证了“多数元素一定存在”。
- **🔑 关键转折点**：
    - **帮派混战类比**：将复杂的计数问题转化为“不同帮派同归于尽，人最多的帮派一定有人剩下”的武侠模型，瞬间理解了 $O(1)$ 空间的精髓。
- **Go 实现细节**：
    - 摩尔投票法在 Go 中实现极其简洁，无需引用任何库。

### [337] 打家劫舍 III (House Robber III)
- **核心思想**：**树形 DP 与状态汇报**。将原本线性的决策搬到了树结构上，父子节点不能同时被偷。
- **解法套路**：DFS 遍历，每个节点向上返回一个“二元组状态”。
- **💡 我的特定卡点**：
    - **语义映射**：最初不理解返回数组 `[0, 1]` 的具体含义，需要建立“索引 0 = 不偷，索引 1 = 偷”的契约意识。
    - **逻辑锁**：困惑于“偷了爸爸为什么儿子只能选索引 0”，后理解这是题目规则强制要求的状态锁定。
- **🔑 关键转折点**：
    - **汇报机制**：意识到如果只给爷爷一个“最大值”，爷爷无法判断是否与爸爸冲突。每个节点必须汇报“偷与不偷”两种情况。
- **Go 实现细节**：
    - 使用 `[]int{doNotRob, doRob}` 进行多值返回，或者在地道 Go 中使用 `(int, int)` 双返回值以提升性能。
- **关联题目**：[968] 监控二叉树 (更复杂的树形 DP)

## 🔗 链表专题

### [206] 反转链表 (Reverse Linked List)
- **核心思想**：**翻多米诺骨牌**。三指针平推（prev, curr, next），每一步掰断一个箭头指向反方向。
- **解法套路**：迭代（三指针） / 递归（信任子问题）。
- **💡 我的特定卡点**：
    - **递归思路**：最初只理解迭代，后悟到递归的本质是"假设后面已经反转完了，我只负责把当前节点挂到末尾"——`head.Next.Next = head; head.Next = nil` 这行是精髓。
- **🔑 关键转折点**：
    - 三指针模型：将操作分解为"记住下家 → 掰箭头 → 右移"三步，每一步职责清晰。
- **Go 实现细节**：
    - `var prev *ListNode` 初始为 nil，正好作为反转后的尾节点指向。
- **关联题目**：[92] 反转链表 II, [25] K个一组翻转

### [21] 合并两个有序链表 (Merge Two Sorted Lists)
- **核心思想**：**两串珠子穿成一条线**。每次比较两条链表头部，挑小的那颗穿进结果链表。
- **解法套路**：哨兵节点（dummy node）统一操作逻辑。
- **💡 我的特定卡点**：
    - **dummy 和 cur 分工踩坑**：实际写代码时把 `cur.Next = list1` 误写成 `dummy.Next = list1`，导致每次拼接都往 dummy 后面接，前面的节点全丢了。
    - **根本原因**：dummy 是锚（永远指向链表头），cur 是针尖（指向最后一个节点）。拼接用 cur，取结果用 dummy。
- **🔑 关键转折点**：
    - dummy = 📌 图钉（不动），cur = 🪡 针尖（持续后移），脚手架类比——盖楼时用它对齐，盖好拆掉。
- **Go 实现细节**：
    - `dummy := &ListNode{}` 创建哨兵，最后 `return dummy.Next` 拆掉脚手架。
- **关联题目**：[23] 合并K个升序链表, [148] 排序链表

### [141] 环形链表 (Linked List Cycle)
- **核心思想**：**操场跑步套圈**。🐢一次 1 步，🐰一次 2 步，有环则兔子必追尾乌龟。
- **解法套路**：快慢指针（Floyd 判圈算法）。
- **💡 我的特定卡点**：
    - **为什么一次 2 步不会跨过慢指针**：相对速度是每轮 1 步，距离每次减 1，必然在某个时刻归零。
- **🔑 关键转折点**：
    - 循环条件必须是 `fast != nil && fast.Next != nil`——fast 一次走两步，两步都安全才敢迈。
- **关联题目**：[142] 环形链表 II

### [142] 环形链表 II (Linked List Cycle II)
- **核心思想**：**两阶段追尾**。第 1 阶段快慢指针找相遇点，第 2 阶段一人回起点同速走，再遇即入口。
- **解法套路**：Floyd 算法两阶段 + 数学推导。
- **💡 我的特定卡点**：
    - **为什么第二阶段一定在入口相遇**：数学推导出 `a = c`（起点到入口 = 相遇点绕回入口），两人同速走 a 步必同时到达入口。
- **🔑 关键转折点**：
    - 手动模拟了 `①→②→③→④→⑤→⑥→③` 的例子，亲眼见证两阶段指针在入口③碰撞。
- **关联题目**：[287] 寻找重复数 (数组中的环形链表)

### [160] 相交链表 (Intersection of Two Linked Lists)
- **核心思想**：**我走过你走的路**。pA 走完 A 跳到 B 起点，pB 走完 B 跳到 A 起点，总路程同为 `a+b+c`，必在交点相遇。
- **解法套路**：双指针交叉走。
- **💡 我的特定卡点**：
    - **无交点时怎么办**：两人各走 `a+b` 步后同时到达 nil，`pA == pB == nil`，返回 nil。
- **🔑 关键转折点**：
    - 理解了 `a+c+b = b+c+a` 这个路程等式是算法的核心保证——不管有没有交点，两人走到终点时各自走过的总距离相等。
- **关联题目**：[1650] 二叉树的最近公共祖先 III

### [19] 删除链表的倒数第 N 个结点 (Remove Nth Node From End)
- **核心思想**：**n 格尺子量尾部**。先拉开 n 格距离，然后尺子平移到尾部，尺尾指向的节点就是待删的前一个。
- **解法套路**：双指针 + dummy。
- **💡 我的特定卡点**：
    - **dummy 开头写成 head**：`dummy := head` 不是真哨兵，必须 `dummy := &ListNode{Next: head}`。
    - **删头节点的 panic**：如果 dummy 不是独立新节点，fast 走 n 步后变成 nil，访问 `nil.Next` 直接崩溃。
- **🔑 关键转折点**：
    - 循环条件是 `fast.Next != nil` 而非 `fast != nil`——因为 slow 要停到待删节点**前一个**，fast 到最后一个节点就该停。
- **关联题目**：[剑指 Offer 22] 链表中倒数第 k 个节点
